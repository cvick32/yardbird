#+title: Prophecy and History Variable Synthesis Implementation Plan

* Scope

Implement auxiliary-variable synthesis for Yardbird's *VMT array mode*.

The first implementation will:

- run only for VMT inputs
- run only with the array abstract strategy
- use existing conflict-scheduler output as the synthesis source
- install auxiliary variables incrementally into the active ~SMTProblem~
- record enough metadata to reconstruct an instrumented abstract VMT model later
- keep synthesis disabled unless explicitly configured

Do not implement SMTLIB support, list support, bv-list support, IC3IA
integration, or original native-array VMT rewriting in the first pass.

* Final Runtime Architecture

Auxiliary synthesis should *not* rebuild the solver as the default path.

The runtime path is:

1. ~ArrayConflictScheduler~ emits normal ground instantiations plus rich conflict
   records.
2. The array strategy classifies conflicts by frame locality.
3. A configurable trigger policy decides whether synthesis fires.
4. If synthesis fires, Yardbird creates one or more ~AuxiliarySpec~ values.
5. ~SMTProblem::install_auxiliary_specs~ declares the new variables for every
   already-created BMC frame.
6. ~SMTProblem::install_auxiliary_specs~ asserts all already-applicable
   auxiliary init/transition/property constraints.
7. Future calls to ~SMTProblem::unroll~ automatically declare/assert the new
   auxiliary constraints at new frames.
8. The proof loop continues at the same BMC depth without rebuilding Z3.

The export path is separate:

1. ~SMTProblem~ stores every installed ~AuxiliarySpec~.
2. ~ProofLoopResult~ includes auxiliary synthesis records.
3. If a VMT dump or final instrumented model is needed, apply the recorded specs
   to the abstracted ~VMTModel~ at the end.

* New CLI Options

Add these options to ~YardbirdOptions~ in ~src/lib.rs~.

Required first-pass options:

- ~--synthesis-trigger <MODE>~
  - default: ~off~
  - accepted modes:
    - ~off~
    - ~detect~
    - ~non-local~
    - ~manual-after-n~
    - ~refinement-limit~
    - ~repeated-pattern~
- ~--synthesis-guard-policy <POLICY>~
  - default: ~true~
  - accepted policies:
    - ~true~
    - ~axiom-local~
    - ~interpolant~
    - ~llm~
- ~--synthesis-after <N>~
  - used by ~manual-after-n~
- ~--synthesis-refinement-limit-window <N>~
  - used by ~refinement-limit~
- ~--synthesis-repeated-pattern-threshold <N>~
  - used by ~repeated-pattern~

Do not add a separate ~--enable-aux-synthesis~ flag. ~off~ means disabled.

Mode behavior:

- ~off~
  - no detection, no synthesis
- ~detect~
  - run detection and logging only
  - never install auxiliary variables
- ~non-local~
  - synthesize when a selected conflict is non-local
- ~manual-after-n~
  - synthesize after N ordinary refinement attempts at a depth
- ~refinement-limit~
  - synthesize when the current refinement step is within N of ~n_refines~
- ~repeated-pattern~
  - synthesize when equivalent or near-equivalent non-local conflicts recur

** Installation algorithm

For each ~AuxiliarySpec~:

1. Convert ~HistorySpec~ into a ~Variable~:
   - current declaration ~H~
   - next declaration ~H_next~
   - VMT-style next relationship command
2. Convert ~ProphecySpec~ into a stuttering ~Variable~:
   - current declaration ~P~
   - next declaration ~P_next~
   - VMT-style next relationship command
3. Push both variables into ~self.variables~.
4. Push current names into ~self.bmc_builder.current_variables~.
5. Insert next-name to current-name mappings into
   ~self.bmc_builder.next_variables~.
6. Register declarations for frames ~0..=self.depth~ in ~Z3VarContext~.
7. For history:
   - build unindexed transition term ~(= H_next (ite guard capture_term H))~
   - store in ~auxiliary_transition_assertions~
8. For prophecy:
   - build unindexed transition term ~(= P_next P)~
   - store in ~auxiliary_transition_assertions~
9. Store localized axiom/property constraints in the appropriate aux assertion
   vectors.
10. Assert retroactive auxiliary constraints:
    - init constraints at frame 0
    - transition constraints for every already-unrolled transition
      ~0 -> 1~ through ~self.depth - 1~
    - do not permanently assert property constraints; they are pushed in
      ~push_property~
11. Register auxiliary subterms in ~SubtermHandler~.
12. Append ~AuxiliaryRecord~.

** Retroactive declaration details

Add helper:

- ~declare_aux_variable_frames(&mut self, variable: &Variable, max_depth: u16)~

For each frame ~0..=max_depth~:

- set ~bmc_builder.depth = frame~
- index the variable's current declaration
- accept it into ~z3_var_context~

Restore the original BMC depth afterward.

For transition constraints over existing frames:

- for each transition frame ~i~ in ~0..self.depth~
  - set ~bmc_builder.depth = i + 1~
  - use ~index_transition_term~
  - assert into Z3

Be careful: ~index_transition_term~ temporarily subtracts one from builder depth.
Do not call it at depth 0.

** Future unroll behavior**

Update ~SMTProblem::unroll~ so after original variables/assertions are added for
a new depth, it also:

1. declares all auxiliary variables for the new frame
2. asserts all auxiliary transition constraints for the new transition
3. keeps auxiliary property constraints out of the solver until ~push_property~

** Property push behavior**

Update ~push_property~.

Current behavior:

- assert ~(not original_property_at_depth)~

New behavior:

- build ~(and original_property_at_depth aux_property_1_at_depth ...)~
- assert its negation

If there are no aux property assertions, preserve current behavior exactly.

** Subterm handling**

Extend ~SubtermHandler~ with:

- ~register_auxiliary_terms(init_terms, transition_terms, property_terms)~

This should:

- extract non-boolean subterms
- update read/write metadata
- include aux transition/property terms in future cost-function input

* Array Strategy Changes

Modify ~src/strategies/array_abstract.rs~.

** Strategy fields**

Add to ~Abstract<F>~:

- ~aux_config: AuxSynthesisConfig~
- ~aux_trigger_state: AuxTriggerState~
- ~aux_records: Vec<AuxiliaryRecord>~

Construct these from ~YardbirdOptions::build_array_strategy~.

** SAT path**

After ~saturate_with_array_types~:

1. receive ~ArrayConflictRecord~ values in addition to existing outputs
2. update trigger state with depth/refinement/conflict metadata
3. if trigger is ~off~, do nothing
4. if trigger is ~detect~, log detection and continue normal ground refinement
5. if trigger fires:
   - select one conflict
   - synthesize guard using configured guard policy
   - build ~AuxiliarySpec~
   - store it in ~ArrayRefinementState~
   - return normal ~ProofAction::Continue~

Do not add a new ~ProofAction~ for synthesis.

** Finish path**

Update ~Abstract::finish~:

1. if state contains auxiliary specs:
   - call ~smt.install_auxiliary_specs(specs)~
   - append returned records to strategy records
2. still add ordinary ground instantiations unless the chosen synthesis policy
   explicitly suppresses the covered instantiation

First pass:

- do not suppress covered ground instantiations
- install aux variables and keep the regular refinement

Later optimization:

- if a localized axiom fully covers the non-local ground instance, skip the
  original ground instance

** Result path**

Update ~Abstract::result~:

- include strategy auxiliary records in ~ProofLoopResult~
- also include any records stored on ~SMTProblem~

* Trigger Implementation

Location:

- ~src/auxiliary_synthesis/trigger.rs~

Create:

- ~AuxTriggerState~
- ~TriggerDecision~

~TriggerDecision~ fields:

- ~fired: bool~
- ~reason: String~
- ~selected_conflict_id: Option<String>~
- ~detected_conflicts: Vec<String>~

Policies:

** ~Off~

- never inspect conflicts deeply
- never fire

** ~Detect~

- identify non-local and repeated-pattern candidates
- log detection
- never fire

** ~NonLocal~

- fire on the first selected non-local conflict
- prefer regular conflicts over high-cost/const conflicts

** ~ManualAfterN~

- fire if ~refinement_step >= N~ and a non-local conflict exists

** ~RefinementLimit~

- fire if ~n_refines - refinement_step <= window~ and a non-local conflict
  exists

** ~RepeatedPattern~

- normalize conflict terms by replacing concrete frame numbers with relative
  offsets
- count normalized patterns across depth/refinement steps
- fire when count reaches threshold

* Guard Synthesis

Location:

- ~src/auxiliary_synthesis/guard.rs~

Create:

- ~synthesize_guard(policy, conflict, smt, model) -> Term~

Policies:

** ~True~

Return ~true~.

** ~AxiomLocal~

Extract local Boolean predicates from the selected conflict:

- equalities
- disequalities expressed as ~(not (= ... ...))~
- arithmetic comparisons

Keep only predicates whose frame span is local to the capture frame.

First pass:

- if extraction is uncertain, return ~true~

** ~Interpolant~

Requires working ~SMTProblem::to_smtinterpol~.

First implementation may be stubbed behind a clear error or fallback:

- if interpolants unavailable, log and fall back to ~axiom-local~

Later implementation:

- extract atomic predicates and negations from sequence interpolants
- filter by locality and model truth

** ~Llm~

Reach goal only.

First implementation:

- enum variant exists
- returns an explicit "not implemented; falling back to true" log message

Later:

- pass conflict, local vocabulary, property terms, and candidate interpolant
  atoms to an external guard proposer

* Auxiliary Spec Construction

Location:

- ~src/auxiliary_synthesis/spec.rs~

Create:

- ~build_auxiliary_spec(conflict, guard, depth, refinement_step, trigger, guard_policy)~

First-pass construction:

1. Select one captured term from the non-local conflict.
2. Prefer non-array scalar terms whose frame is outside the chosen local
   transition window.
3. Create a history variable for that captured term.
4. Create a stuttering prophecy variable with the same sort.
5. Replace the far-frame occurrence in the localized axiom with the prophecy
   variable.
6. Add a property or auxiliary constraint connecting prophecy/history where
   needed.

If reliable replacement is too large for the first patch, split implementation:

- Phase A: install stuttering prophecy/history variables and log records
- Phase B: add localized axiom replacement

Sort inference order:

1. existing variable declarations
2. typed array function names
3. Z3 AST sort if reachable through ~SMTProblem~
4. explicit unsupported error

* SMTInterpol Work

Interpolation is not required for the first true-guard runtime synthesis pass,
but guard policy ~interpolant~ needs it.

Modify:

- ~src/smt_problem.rs::to_smtinterpol~
- ~src/utils.rs::run_smtinterpol~

Implementation requirements:

- stop using fixed global ~interpolant-out.smt2~
- use a temporary file
- generate named partitions for:
  - init
  - each transition frame
  - negated property
- include auxiliary assertions in the appropriate partitions
- parse SMTInterpol output without assuming all interpolants are on one line

Recommended implementation:

- make runtime ~SMTProblem~ retain structured indexed assertion terms as they
  are asserted
- emit those retained terms for interpolation

Do not reconstruct from ~VMTModel~ for the runtime interpolation path unless the
incremental assertion log proves too awkward.
