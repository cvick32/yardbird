#+title: Fault Localization Prototype Plan for Yardbird

* Purpose

This plan describes a small, implementation-oriented prototype for using
Yardbird as a backend for consistency-based software diagnosis and repair-aware
benchmarking.


1. run a few ~h~-guarded fault-localization encodings through Yardbird,
2. compute minimal sets of source lines / repair sites whose health assumptions
   must be relaxed,
3. expose Yardbird's array instantiation and refinement data for those
   diagnoses,
4. ingest Tool's's benchmark metadata and reference outputs so Yardbird runs
   can be compared against them,
5. clarify how much of the CBMC-to-SMT2 path works before bit-blasting.

The motivating idea is that Tool's's WCNF/CNF pipeline gives line-number
diagnoses, but Yardbird can additionally expose *which array instantiations and
refinement choices made those diagnoses possible*. That is the part that could
become a genuinely Yardbird-specific research direction.

* Core Thesis

Tool's encodes software diagnosis by adding Boolean health variables to the
bounded encoding of a program. A health variable ~H_i~ says that source statement
or repair site ~i~ behaves according to the original program. Setting all
~H_i = true~ should make the failing test plus expected postcondition
unsatisfiable. Relaxing a small set of ~H_i~ variables asks whether changing
those source lines could make the property hold.

Yardbird can use the same diagnosis idea, but at SMT/VMT level instead of
bit-blasted WCNF:

- hard constraints: program semantics, test inputs, expected postcondition,
  repair-model constraints
- soft / assumption constraints: ~H_i~ for each repairable statement
- diagnosis: a subset-minimal set of health variables to omit/relax so the
  formula becomes satisfiable
- Yardbird-specific signal: array axiom instantiations, refinement steps,
  unsat-core membership, cost-function decisions, and SAT models

* Prototype Scope

The first prototype should be intentionally narrow.

Include:

- SMT-LIB inputs first
- array theory first
- single-file benchmark harness
- health variables with names compatible with Tool's:
  ~compHealthy::<file>::<line>~
- single-test diagnosis first
- multiple-test handling after the single-test loop works
- core-guided diagnosis search outside Z3/Yardbird
- JSON result output compatible enough to compare with Tool's's
  ~solutions~ and ~aggregatedSolutions~

Defer:

- source-code repair synthesis
- pointer repair models
- full C memory model support
- floats and varargs-heavy C-Pack programs
- TCAS import unless SIR data is available
- a high-performance HSD implementation
- learned cost functions
- replacing Tool's end-to-end

Initial examples:

2. ~array_bounds~

   - exercises Yardbird array abstraction
   - repair-model constraint should prevent the invalid "set index out of

3. optional ~dead_code~
   - useful for checking the paper's warning about optimizations and source
     structure

** Files To Add

Create:

- ~examples/smtlib/fault-localization/maxplusone.smt2~
- ~examples/smtlib/fault-localization/array_bounds.smt2~
- ~examples/smtlib/fault-localization/dead_code.smt2~
- ~examples/smtlib/fault-localization/README.md~

Keep the encodings small and readable. Do not try to mirror CBMC's full SSA

** Encoding Pattern

For an assignment:

#+begin_src smt2
(declare-const compHealthy::example.c::24 Bool)
(assert (=> compHealthy::example.c::24 (= x_2 (+ a_1 1))))
(assert (=> (not compHealthy::example.c::24) repair_model_for_x_2))
#+end_src

For a branch condition:

#+begin_src smt2
(declare-const compHealthy::example.c::20 Bool)
(declare-const branch_20 Bool)
(assert (=> compHealthy::example.c::20 (= branch_20 (< a_1 b_1))))
#+end_src

For initial prototype queries, health variables can be handled as assumptions
from the diagnosis engine rather than asserted directly in the SMT-LIB file.

** Validation

Run:

- Yardbird concrete SMT-LIB mode if possible
- Yardbird abstract SMT-LIB mode for the array example
- Z3 directly as a sanity check if Yardbird parsing blocks on syntax

Expected output:

- all-health query is UNSAT for failing examples
- relaxing expected diagnoses makes the formula SAT
- relaxing invalid diagnoses remains UNSAT when repair-model constraints are
  present

* Milestone 1: Add A Minimal Diagnosis Engine In Yardbird

** Goal

Implement a small Rust-side core-guided diagnosis loop over health variables.
This is the replacement/prototype for Tool's's Python + HSD stage.

** New Module Layout

Create:

- ~src/fault_localization/mod.rs~
- ~src/fault_localization/config.rs~
- ~src/fault_localization/health.rs~
- ~src/fault_localization/query.rs~
- ~src/fault_localization/core.rs~
- ~src/fault_localization/hsd.rs~
- ~src/fault_localization/result.rs~

Responsibilities:

- ~config.rs~
  - CLI and runtime config
- ~health.rs~
  - health variable discovery and source-location parsing
- ~query.rs~
  - solver query construction and assumption management
- ~core.rs~
  - project unsat cores to health variables
  - optionally shrink cores by deletion
- ~hsd.rs~
  - minimal hitting-set / diagnosis search
- ~result.rs~
  - JSON output compatible with Tool's enough for comparison

** CLI Options

Add options to ~YardbirdOptions~:

- ~--fault-localization~
  - enable diagnosis mode
- ~--health-prefix <PREFIX>~
  - default: ~compHealthy::~
- ~--diagnosis-limit <N>~
  - optional cap on number of diagnoses
- ~--minimize-cores~
  - deletion-minimize health-variable cores before feeding HSD
- ~--diagnosis-json-output <PATH>~
  - optional file output, distinct from existing proof JSON

Keep these options disabled by default.

** Initial Algorithm

Use a simple Reiter/HSD-style loop.

Definitions:

- ~H~ = set of health variables
- ~D~ = candidate diagnosis, meaning these health assumptions are relaxed
- query ~(H \ D)~ = solve with all health variables except ~D~ asserted true

Algorithm:

1. Query with all health assumptions enabled.
2. If SAT, return ~allHealthy~.
3. If UNSAT, extract a core projected to health variables.
4. Maintain a set of discovered health cores.
5. Enumerate subset-minimal hitting sets of discovered cores.
6. For each candidate hitting set ~D~:
   - query with ~D~ relaxed
   - if SAT, emit ~D~ as a diagnosis
   - if UNSAT, add the projected core and continue
7. Stop when no new minimal hitting sets remain or ~--diagnosis-limit~ is hit.

This does not need to be maximally optimized in V1. The design should make it
easy to replace the enumerator later.

** Solver Interface Needs

The diagnosis engine needs:

- a way to assert health variables as assumptions or tracked assertions
- a way to retrieve an unsat core over those assumptions
- a way to run repeated SAT checks with different health relaxations
- a way to preserve or replay Yardbird-generated array instantiations

Implementation options:

1. use Z3 assumptions directly in a new local API around Yardbird's SMTLIB
   solver path;
2. use push/pop plus ~assert_and_track~ labels;
3. for the first manual corpus, regenerate a fresh solver per candidate query
   and prioritize clarity over performance.

Prefer option 3 for the first few days if it avoids invasive changes.

* Milestone 3: Patch CBMC Fork To Emit Guarded SMT2

** Goal

Modify the Tool's CBMC fork enough to emit SMT2 formulas that include the
same health-variable guarded assignment constraints currently used for WCNF.

** Proposed CBMC Change

Add a flag such as:

- ~--fault-loc-instrument~

Semantics:

- enables health-variable instrumentation of repairable assignments
- enables observation handling
- does *not* force WCNF
- can be combined with ~--smt2 --outfile <file>~
- can still be combined with ~--wcnf~ for compatibility if desired

Implementation approach:

1. Split the current WCNF boolean through the symex equation conversion layer:
   - current: ~wcnfIsSet~
   - proposed: ~faultLocInstrument~
2. Rename/generalize:
   - ~convert_assignments_wcnf~ -> ~convert_assignments_fault_loc~
   - ~convert_wcnf_assignment_step~ -> ~convert_fault_loc_assignment_step~
   - ~convert_observation_step~ can keep WCNF names internally at first, but
     should eventually use neutral names.
3. In solver selection, allow SMT2 to be selected when
   ~--fault-loc-instrument --smt2~ is passed.
4. Preserve source-location-compatible health variable names:
   ~compHealthy::<file>::<line>~.
5. Add a mapping dump for health variables in SMT2 output if CBMC's symbol
   names become escaped or difficult to recover.

** Expected Command

cbmc example_sherloc.c \
  --fault-loc-instrument \
  --smt2 \
  --outfile example_faultloc.smt2 \
  --no-propagation \

#+end_src

* Milestone 6: Repair-Aware Cost Function Experiments

** Goal

Use fault-localization metadata to bias Yardbird's array instantiation choices
toward terms that are likely to expose small diagnoses.

** New Cost Function

Add:

- ~src/cost_functions/array/repair_aware.rs~

Inputs:

- current BMC depth
- normal transition/property vocab
- reads/writes
- health variable metadata
- optional current candidate diagnosis or unsat health core
- optional source-line-to-term ownership map

Initial scoring ideas:

- discount terms appearing in postcondition/assertion vocabulary
- discount array indices appearing near repair sites in the current health core
- discount reads/writes whose SSA/source location maps to suspicious lines
- penalize terms unrelated to any health core
- keep the existing "avoid pc" behavior from current BMC/index-aware costs

V1 can be heuristic. The point is to see whether diagnosis-aware term selection
reduces refinement steps or produces smaller/more stable cores.

* Milestone 7: Decide How Far To Push CBMC SMT2

** Goal

After 1-2 days of experiments, classify CBMC SMT2 output into buckets.

Buckets:

1. Yardbird parses and solves directly.
2. Yardbird parses after small SMT-LIB parser/Z3VarContext fixes.
3. Yardbird needs theory support we can reasonably add.
4. CBMC output is too memory-model-specific or bit-vector-heavy for this path.

Likely blockers:

- bit-vector operations not currently covered in Yardbird's SMT-LIB path
- arrays indexed by bit-vectors instead of ints
- C memory objects encoded as structs/datatypes
- floating-point theory from C-Pack examples
- varargs wrappers in instrumented C-Pack programs
- pointer checks and bounds checks generating formulas outside Yardbird's
  current supported fragment

Document every unsupported construct with:

- the benchmark that triggered it
- the smallest extracted SMT2 snippet
- whether Z3 accepts it
- whether adding support to Yardbird is worth it

* Open Design Questions

- Should Yardbird's diagnosis mode live in the main binary or in a separate
  experimental binary under ~src/bin/~?
- Should the first solver loop rebuild a fresh solver per candidate diagnosis,
  or should it push/pop assumptions on a persistent solver?
- Should array instantiations be cached across diagnosis candidates?
- Should health variables be represented as assumptions, tracked assertions, or
  ordinary symbols asserted through push/pop?
- Should CBMC emit one SMT2 file plus a metadata sidecar, or should all mapping
  information be embedded in SMT-LIB comments?
- How much CBMC SMT2 support should Yardbird absorb before it stops being a
  clean research prototype?
- Do we want compatibility with Tool's's exact JSON output or a richer
  Yardbird-native result with a Tool's compatibility view?

* Risks And Mitigations

** CBMC SMT2 may be too hard for Yardbird

Mitigation:

- start with manual SMT-LIB examples
- classify blockers precisely
- use CBMC output as a benchmark discovery tool even when Yardbird cannot parse
  it directly
- consider a restricted C-to-VMT/SMT translator for tiny array examples if CBMC
  emits too much memory-model machinery

** WCNF instrumentation assumptions may not transfer cleanly to SMT2

Mitigation:

- patch at the symex equation level, where health implications already exist
- test with ~maxplusone.c~ first
- preserve WCNF behavior while adding a separate ~--fault-loc-instrument~ path

** Minimal diagnosis search may become solver-call dominated

Mitigation:

- implement the simplest correct loop first
- add core minimization only behind a flag
- cache parsed problems and, later, solver states
- reuse Yardbird instantiations across related queries after correctness is
  established

** Source-line mapping may be fragile

Mitigation:

- use Tool's's ~compHealthy::<file>::<line>~ naming convention
- emit explicit sidecar metadata from CBMC if SMT2 escaping makes names awkward
- compare line-number sets against Tool's's reference JSON early

** Repair model constraints may be underspecified

Mitigation:

- first support the general repair model
- add bounds-safe array repair constraints manually for ~array_bounds~
- treat pointer-safe repairs as a later phase

* Definition Of Done For The First Prototype

The prototype is successful if we can show:

1. a manual ~h~-guarded SMT-LIB array example runs through Yardbird,
2. Yardbird reports at least one minimal diagnosis matching the expected line
   set,
3. the result includes Yardbird instantiation/refinement metadata,
4. Tool's C-Pack reference data is imported into a manifest,
5. a patched or clearly specified CBMC path can emit guarded SMT2 for at least
   one tiny example, or the blockers are documented with exact source locations
   and SMT-LIB snippets.

That is enough to decide whether this direction should become a real Yardbird
feature, a benchmark ingestion project, or a paper-experiment side branch.
